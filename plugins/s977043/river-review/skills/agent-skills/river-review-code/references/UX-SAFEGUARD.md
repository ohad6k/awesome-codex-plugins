# UX-SAFEGUARD 観点 — 操作の安全装置レビュー

Nielsen ヒューリスティクス（エラー防止・ユーザーコントロール・回復支援）のうち、**diff から静的に検出できる部分**に限定した2観点。視覚評価・ユーザビリティテストの代替ではなく、コード上の安全装置の欠落を検出する。

River Review の契約に従い **report-only** である。fix は「次の最小一手」の提案テキストとして出力し、適用はしない。

## Pre-execution Gate

**最初に判定する**。満たさない場合は以降のチェック（Grep 含む）を行わず `NO_REVIEW` を返す。

- 差分に**ユーザー操作に反応するコード**が含まれる: 削除・上書き・送信・公開などのハンドラ / mutation 呼び出し / フォーム・入力バリデーション / ユーザー向けエラーメッセージ文字列のいずれか
- docs・設定ファイル・テストのみの差分、バックエンドのバッチ処理のみの差分は対象外
- ビルド成果物・生成物（`dist/**`・`*.map`・lockfile・自動生成 manifest）は除外する

## 観点 / Perspectives

| 観点                      | 対象とする欠落                                     | 核心の問い                                               |
| ------------------------- | -------------------------------------------------- | -------------------------------------------------------- |
| **Destructive Safeguard** | 不可逆操作の確認・取り消し手段の欠如               | 「この操作、**誤って実行したら**戻れる？」               |
| **Error Recovery**        | 回復方法を示さないエラーメッセージ・失敗の握り潰し | 「失敗したとき、ユーザーは**次に何をすべきか**わかる？」 |

## 委譲 / Delegation（既存スキル優先）

重複指摘を避けるため、以下の領域は既存 registry skill に委譲する。本観点は**委譲先が扱わない残余のみ**を検出する。

| 対象領域                                  | 委譲先                                                 | 本観点が扱う残余                                       |
| ----------------------------------------- | ------------------------------------------------------ | ------------------------------------------------------ |
| DB スキーマ・マイグレーションの破壊的変更 | `migration-safety` / `laravel-migration-safety`        | UI・API 層のユーザー操作に対する安全装置               |
| mutation の認証・認可                     | `nextjs-server-action-security` / security 系観点      | 認可済みユーザー自身の誤操作からの保護                 |
| loading / error / empty state の表示欠落  | `loading-state`                                        | エラー表示は存在するが**回復方法を示さない**文言       |
| コンポーネントの状態定義（disabled 等）   | `component-variants-states`                            | 状態は定義済みだが破壊的操作に確認ステップがないケース |
| フォームの a11y（ラベル・live region）    | `a11y-accessible-name` / `modern-web-a11y-interactive` | a11y ではなく文言・フローとしての回復支援              |
| React Router action のエラー表示契約      | `react-router-action-contract`                         | フレームワーク非依存の回復支援文言・確認フロー         |

## 各観点のチェック / Checks

### 1. Destructive Safeguard（破壊的操作の安全装置）

- 差分に追加された削除・上書き・公開・送信取り消し不能な操作のハンドラに、確認ステップ（確認ダイアログ・タイプ確認・二段階操作）または取り消し手段（undo・ソフトデリート・猶予期間）が存在するか
- 一括操作（複数件削除・全件更新）は単件操作より高い基準で確認する（影響件数の提示があるか）
- 検出時は**差分内の操作ハンドラ**（`file:line`）にアンカーし、どの安全装置が欠けているかを特定して指摘する
- 証拠要件: 確認 UI・undo 機構が**別ファイルに存在する可能性**を code_search（confirm / dialog / modal / alert / prompt / cancel / undo / restore / rollback / softDelete 等）で棄却できた場合のみ findings とする。棄却できない場合は questions に落とす

### 2. Error Recovery（入力エラーの回復支援）

- 差分に追加されたユーザー向けエラーメッセージが、原因と**次にとるべき行動**を伝えているか（「エラーが発生しました」だけの文言、エラーコードの生表示を検出する）
- フォーム送信の失敗時に入力内容が保持されるか（全消去してやり直させる実装でないか）
- バリデーションエラーが**どのフィールドの何が問題か**を特定できる形で返っているか（全体エラー1つに丸めていないか）
- 検出時は差分内のエラーメッセージ文字列・エラーハンドリング箇所にアンカーする

## False-positive guards

- 指摘行（finding の `file:line`）が差分内にあること。差分外の既存 UI への一般論は禁止（review-policy §3.1 の欠落指摘アンカー規則に従う）
- 開発者向けログ・内部エラー（ユーザーに表示されないもの）は対象外（`logging-observability` の責務）
- 確認レス操作が仕様として意図されている証拠（コメント・PR 説明・undo の存在）があれば指摘しない
- 情報設計・文言の好みの良し悪しは断定しない。回復手段の**有無**のみを扱う
- 観点ごとに最大5件。超過分は severity 降順で切り捨てる

## Output

[SKILL.md](../SKILL.md) の Output Format に従い（Finding / Impact / Fix）、各 finding に Severity と Confidence（`high` / `medium` / `low`）を併記する。severity は**出力スキーマ語彙**（`info` / `minor` / `major`）で書く。データ喪失に直結する Destructive Safeguard の欠落は `major` を起点、Error Recovery は `minor` を起点とし、確信が持てない場合は `info` に落とす。安全装置が差分外に存在するかを棄却しきれない指摘は findings ではなく questions として返す。
